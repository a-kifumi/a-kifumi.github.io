# Repository instructions

This repository publishes the `a-kifumi.github.io` portfolio in English and Japanese.

## Mandatory project-addition gate

Before adding, removing, renaming, reordering, or creating a page for any project, read [`docs/project-addition-policy.md`](docs/project-addition-policy.md) in full and apply every required check. Do not edit project listings first and read the policy afterward.

The portfolio may support discovery and naturally earned references, but its primary purpose is to help visitors understand real projects by akifumi. Domain Rating, link equity, keyword rankings, or outbound-link volume must never be treated as the primary reason for publishing a project or page.

## Repository rules

- Verify the project from a source of truth: its local repository, production site, App Store page, or other owner-controlled public page.
- Publish only projects made or maintained by akifumi. Never invent status, features, metrics, testimonials, awards, or availability.
- Keep `index.html` and `ja/index.html` aligned unless a language-specific difference is intentional and documented.
- Prefer one useful, natural destination per project card. Add secondary links only when they serve a distinct visitor need.
- Use concise, descriptive anchor text. Do not repeat keyword-rich exact-match anchors to influence rankings.
- Do not create thin, duplicate, doorway, location/keyword-variant, or mass-templated pages.
- Do not add hidden links or text, reciprocal-link schemes, paid links that pass ranking credit, or links whose main purpose is search manipulation.
- A dedicated project page requires original value beyond a card: verified explanation, meaningful visual evidence, and useful project context. If that material is unavailable, keep the project as a card or postpone publication.
- Preserve unrelated user changes. Inspect `git status` before editing and keep the diff limited to the requested project.
- After editing, run the verification checklist in the policy and report any item that could not be verified.

## Instruction priority

User instructions take priority. These repository rules and the project-addition policy apply unless the user explicitly replaces a specific rule. Never weaken the anti-spam requirements merely to increase the number of links or pages.
